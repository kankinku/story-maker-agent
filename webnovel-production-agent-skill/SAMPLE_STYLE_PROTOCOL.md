# Sample Style Protocol

이 문서는 사용자가 제공한 웹소설 샘플을 거대한 문체 프롬프트로 요약하지 않고, 재사용 가능한 제작 구조로 바꾸기 위한 프로토콜이다.

핵심 원칙은 원문 문장과 고유 설정을 복제하지 않고, 원문에서 발견한 **연출 결정의 작동 방식**만 추출하는 것이다.

## 1. 역할 분리

샘플 기반 제작은 네 역할로 나눈다.

1. `style_extractor`: 샘플에서 문체, 정보 채널, 문단 리듬, 반복 비트, 금지 패턴을 추출한다.
2. `scene_planner`: 추출된 규칙을 회차 계약과 장면 비트로 변환한다.
3. `chapter_writer`: 승인된 정본과 장면 계약을 바탕으로 초고를 쓴다.
4. `reviser`: 결과가 규칙을 기능적으로 재현했는지 감사하고 최소 수정안을 낸다.

독립된 최상위 스킬을 늘리기보다, 이 패키지에서는 위 역할을 기존 `Serial Plan`, `Narrative Control`, `Draft Buffer`, `Polish and Guard` 단계에 연결한다.

## 2. 추출 축

샘플 분석은 다음 축을 별도 기록한다.

### 정보 전달 채널

- `narration`: 주인공의 주관적 해석, 위험 평가, 자기합리화.
- `dialogue`: 관계 압력, 정보 교환, 캐릭터 목소리.
- `system`: 객관적 규칙, 보상, 금지 조건, 상태 변화.
- `community`: 해설, 개그, 반론, 복선, 독자 대리 반응.
- `action`: 관찰, 가설, 시도, 결과, 재도전.

중요한 것은 채널의 겉모양이 아니라 기능이다. 시스템창이나 댓글을 넣어도 정보 변화나 행동 압력이 없으면 장면은 정체된다.

### 마이크로 비트

장면 안의 짧은 추진 단위는 기본적으로 다음 순서를 가진다.

1. 상황 제시
2. 주인공의 즉각적 해석
3. 예상 또는 자신감
4. 예상 밖의 반전
5. 짧은 감정 반응
6. 새 행동 목표

모든 장면에 여섯 단계를 기계적으로 넣지 않는다. 빠진 단계가 있으면 의도적인 생략인지, 추진력 누락인지 감사한다.

### 도전 루프

반복 공략형 장면은 다음 상태 전이를 가진다.

```text
관찰 -> 가설 -> 시도 -> 실패 또는 비용 -> 패턴 학습 -> 외부 피드백 -> 수정된 재도전
```

실패가 누적되는데 학습 결과가 남지 않으면 `retry_without_learning`으로 본다.

### 기대-반전-보상

```yaml
id: expectation-reversal-reward
scope: scene
purpose: 독자의 예측을 깨면서 다음 행동 동력을 만든다.
trigger:
  - 주인공이 보상, 성공, 안전을 확신한다.
  - 시스템, 조력자, 적, 댓글이 새 정보를 제시한다.
procedure:
  - 기대를 구체적으로 세운다.
  - 기대를 잠시 확신하게 만든다.
  - 예상보다 하찮거나 위험하거나 비틀린 결과를 제시한다.
  - 짧은 감정 반응을 배치한다.
  - 결과 속에서 다음 공략 가능성이나 새 문제를 발견한다.
avoid:
  - 반전 뒤 감탄만 길게 이어가기.
  - 같은 정보를 서술과 시스템 메시지로 중복 설명하기.
  - 반전이 다음 행동과 무관하게 끝나기.
verify:
  - 장면 종료 시 목표, 판단, 정보, 관계 중 하나 이상이 바뀌었는가.
  - 반전이 다음 행동을 강제하는가.
```

## 3. 회차 계약

원고 작성 전에 회차별로 다음 계약을 만든다.

- 이번 화에서 바뀌어야 할 상태.
- 핵심 기대.
- 핵심 반전.
- 독자가 얻을 보상.
- 화말의 열린 질문 또는 다음 행동 압력.
- 이번 화에 사용할 정보 채널.
- 반복될 장면 비트와 의도적으로 피할 비트.

화말은 단순 중단이 아니라 다음 행동을 강제하는 정보, 비용, 선택지, 관계 변화로 끝나야 한다.

## 4. 장면 비트

장면 카드는 다음 필드를 가진다.

- `goal`: 장면에서 인물이 얻으려는 것.
- `obstacle`: 즉시 막는 힘.
- `judgment`: 주인공이 상황을 어떻게 해석하는지.
- `action`: 실제 선택과 행동.
- `result`: 상태 변화.
- `next_pressure`: 다음 장면으로 밀어내는 압력.
- `channels`: 사용 채널과 각 채널의 서사 기능.
- `micro_beats`: 기대, 반전, 감정 반응, 새 목표.
- `forbidden_moves`: 이번 장면에서 피할 과적합 또는 반복.

## 5. 샘플 스타일 프로필

스타일 프로필에는 고정 성질만 둔다.

- POV와 인식 주체.
- 서술 태도.
- 기본 정서.
- 문단 길이 분포.
- 대화, 서술, 독백, 시스템, 커뮤니티 비율.
- 채널별 기능.
- 도전 루프와 실패 학습 방식.
- 금지 패턴.

샘플의 문장, 농담, 고유명사, 사건 배열을 그대로 저장하지 않는다.

## 6. 감사 기준

정량 검사는 다음을 본다.

- 비어 있지 않은 줄 수.
- 문단 길이 중앙값과 짧은 문단 비율.
- 연속 장문 문단 수.
- 채널 다양성.
- 반복 감탄사와 접속어 밀도.
- 화말 열린 고리 존재.

의미 기반 평가는 다음 1~5점 루브릭을 사용한다.

- `voice_consistency`
- `scene_propulsion`
- `information_naturalness`
- `expectation_and_reversal`
- `comedy_tension_balance`
- `character_distinction`
- `chapter_end_pull`
- `continuity`

LLM 판단만으로 승격하지 않는다. 정량 검사, 정본 일치, 사람 검토를 함께 사용한다.

## 7. 안티패턴

- 모든 문장을 지나치게 짧게 만들어 리듬이 단조로워짐.
- 매 문단마다 감탄이나 자기합리화를 반복함.
- 원문 유행어, 농담, 문장 구조를 그대로 가져옴.
- 시스템창으로 설명한 정보를 서술자가 다시 해설함.
- 댓글이 무작위 농담만 하고 장면 기능이 없음.
- 실패가 반복되지만 관찰이나 학습 결과가 없음.
- 매 화 같은 종류의 클리프행어를 사용함.
- 액션 샘플에만 과적합해 조용한 관계 장면도 빠르고 시끄럽게 씀.

## 8. 적용 순서

1. `calibrate_from_samples.py`로 샘플의 용어, 말투, 문단 구조 후보를 추출한다.
2. `style_profile.json`을 사람이 검토해 active 규칙만 남긴다.
3. `scene_contract.json`으로 회차와 장면 계약을 작성한다.
4. Episode Writer가 계약을 원고로 확장한다.
5. `audit_style_profile.py`와 `audit_lexicon.py --manuscript`를 실행한다.
6. 필요하면 `run_semantic_rubric.py`로 사람 또는 별도 judge가 작성한 1~5 루브릭 점수를 검증한다.
7. 한국어 최종 원고는 `$humanize-korean` 이후 `ai_tell_guard.py --fail-on-s1`을 통과해야 한다.
