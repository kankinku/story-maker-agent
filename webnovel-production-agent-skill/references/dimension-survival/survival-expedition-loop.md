# Survival Expedition Loop

Use this reference for dimension-transfer survival stories whose core pleasure is not travel itself, but timed expeditions that solve a deficit in the home world through farming, conversion, and base growth.

## Core Engine

```text
home-world deficit
-> expedition objective
-> gear and carrying-capacity prep
-> dimension entry
-> time, environment, exit, and threat scan
-> safe route selection
-> resource candidate discovery
-> risk, weight, and utility comparison
-> fight, trade, stealth, or salvage
-> limited resource selection
-> return
-> conversion into survival gear or infrastructure
-> base and combat capability improvement
-> new deficit
```

The protagonist does not become stronger simply because the dimension is new. They improve survival odds by extracting resources, techniques, equipment, and companions, then converting those gains into the home world's survival system.

## Deficit Before Reward

```yaml
id: establish-deficit-before-reward
purpose: Make a small reward feel valuable by proving the shortage first.
procedure:
  - Show the missing resource through physical evidence and body symptoms.
  - Confirm that ordinary methods cannot solve it.
  - State the protagonist's immediate need in one sentence.
  - Make the first reward a minimal resource that directly answers that deficit.
  - Show sensory and bodily relief after use.
avoid:
  - Explaining the deficit without making it felt.
  - Starting with oversized combat power instead of survival relief.
  - Letting the system window explain the reward's value alone.
verify:
  - The reward would feel weak if the deficit scene were removed.
  - The first use changes the protagonist's body, choices, or available time.
```

## Ability Rule

New abilities should create freedom and maintenance cost at the same time.

Examples:

- Future gear -> battery shortage.
- Better carrying capacity -> heavier resources become possible, but route risk rises.
- Technical companion -> information improves, but protection and repair become obligations.
- Mobility platform -> exploration expands, but parts and power become bottlenecks.
- Revisit control -> strategic choice expands, but unresolved threats can return.

## Expedition Episode Modes

### Start Episode

1. Confirm a home-world deficit or new objective.
2. Check available gear and carrying limits.
3. Declare collection priorities.
4. Transfer dimensions.
5. Reveal an environment different from expectation.
6. System announces time, hazard, threat, or exit condition.
7. End on the protagonist's short practical reaction.

### Middle Episode

1. Reconfirm location, time, and resources.
2. Discover a tempting resource or route.
3. Compare cost and risk.
4. Introduce an unexpected obstacle.
5. Use gear or companion capability.
6. Acquire a limited reward.
7. Reveal a larger threat or world secret.

### Exit Episode

1. Calculate remaining distance and time to the exit.
2. Triage secured resources.
3. Add a companion or mission complication.
4. Force conflict between safe return and risky choice.
5. Make the protagonist decide by their survival principle or its deliberate violation.
6. Return or reverse course.
7. End on a concrete action that must be solved next.

### Return/Base Episode

1. Inspect the loot as physical objects.
2. Solve an immediate survival problem.
3. Craft equipment or improve the base.
4. Demonstrate the new tool in use.
5. Reveal a new maintenance cost or missing resource.
6. Prepare the next expedition.

## Cliffhanger Types

- `dimension_reversal`: expected one environment but arrived in another.
- `pursuit_started`: enemy, security system, or infection recognizes the protagonist.
- `forced_choice`: return safely or rescue/secure something at cost.
- `trade_offer`: local actor offers survival through a deal.
- `world_secret`: cause of collapse or hidden designer is revealed.
- `emotional_failure`: victory is followed by injury, loss, or unresolved rescue.

Avoid repeating the same cliffhanger type twice in a row.
